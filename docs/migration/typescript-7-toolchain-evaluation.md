# TypeScript 7 Toolchain Evaluation

This document tracks the current repository-level TypeScript 7 toolchain position after promoting the default workspace compiler to TS7, while keeping `ttsc` as a focused secondary validation lane.

The goal is narrower than “replace everything with `ttsc`”:

1. Keep the default workspace compiler on TS7 without regressing package, CLI, or docs flows.
2. Measure whether the isolated `ttsc` lane gives useful local or CI feedback speed.
3. Capture what still blocks `ttsc` from replacing the default root lane.

## Current Position

Today the repository uses TS7 in the default root `typecheck` flow, and it also keeps an isolated `ttsc` lane for comparison and fast no-emit validation.
That is enough to make TS7 the normal compiler line, but not enough to replace the default root `typecheck` flow with `ttsc` yet.

The main reasons are:

- the root lane still does more work than a pure `--noEmit` pass
- the root lane still owns declaration emit plus CLI generation work
- Biome still covers a broader file surface than a TS-only compiler lint path

## Commands

Primary commands:

```bash
bun run typecheck
bun run typecheck:ttsc:ts7
bun run typecheck:ttsc:ts7:install
bun run typecheck:ttsc:ts7:clean
bun run graph:ttsc:ts7
bun run benchmark:ttsc:ts7
```

Compatibility aliases remain for the earlier naming:

```bash
bun run typecheck:ttsc:exp
bun run typecheck:ttsc:exp:install
bun run typecheck:ttsc:exp:clean
```

`graph:ttsc:ts7` passes through to `ttsc-graph`, so the same entrypoint can be used for MCP stdio, JSON dump, or the local graph viewer:

```bash
# MCP stdio server
bun run graph:ttsc:ts7

# JSON graph dump
bun run graph:ttsc:ts7 -- dump --tsconfig tsconfig.json > /tmp/k-msg.graph.json

# Local viewer
bun run graph:ttsc:ts7 -- view --tsconfig packages/provider/tsconfig.json
```

## Scope

The isolated lane intentionally covers:

- `packages/core`
- `packages/template`
- `packages/provider`
- `packages/messaging`
- `packages/analytics`
- `packages/channel`
- `packages/webhook`
- `packages/k-msg`
- `apps/cli`
- `apps/docs-hono`

It intentionally does not typecheck the source-content workspace under `apps/docs`, because that directory now holds Markdown, generated inputs, and snippet sources rather than an executable site app.
The Hono docs runtime is part of the TS7 validation surface.

## What The Numbers Mean

Use the benchmark snapshot as a repository-specific signal, not as a generic “TS7 is always X times faster” claim.

Important caveats:

- `bun run typecheck` includes the current `build:types` fan-out and CLI runtime generation before the CLI `tsc --noEmit` pass.
- `bun run typecheck:ttsc:ts7` is a pure `--noEmit` validation lane over selected tsconfigs.
- `ttsc-graph dump` measures graph construction cost for exploration, not a production CI replacement.
- the first runner install cost is real, so the snapshot includes it separately instead of hiding it.

## Practical `ttsc-graph` Workflow

Use `ttsc-graph` in three different modes depending on the question:

- MCP stdio: `bun run graph:ttsc:ts7`
  Use this when an agent or tool wants a live graph server over stdio.
- JSON dump: `bun run graph:ttsc:ts7 -- dump --tsconfig tsconfig.json > /tmp/k-msg.graph.json`
  Use this for repeatable repository snapshots or when you want to diff graph structure offline.
- Local viewer: `bun run graph:ttsc:ts7 -- view --tsconfig packages/provider/tsconfig.json`
  Use this when you want a focused package view instead of the whole workspace graph.

Practical default:

- start with a package-level `--tsconfig` for focused questions
- use the root `tsconfig.json` only when the flow crosses package boundaries
- treat the checked-in benchmark snapshot as the current startup-cost reference for graph usage

## Why This Is Worth Keeping

The evaluation lane already gives three useful signals:

- fast secondary validation for the package and CLI TypeScript surface
- a direct comparison point against the default TS7 compiler lane
- a place to validate `ttsc`-hosted tooling, including `ttsc-graph`, without disturbing the default workflow

## Why `ttsc` Should Not Replace The Default Lane Yet

### The default compiler line still owns more than `--noEmit`

The docs runtime itself is no longer blocked on Astro/TypeDoc, and the workspace default compiler is already on TS7.
However, the default lane is still responsible for:

- declaration emit for publishable packages
- CLI generation that is paired with type validation in the current scripts
- API Extractor compatibility in the checked-in toolchain
- any remaining downstream tool assumptions tied to the default compiler version

That means the repository can keep using `ttsc` as a strong secondary lane today, while the default compiler path stays responsible for the broader packaging workflow.

### Biome still covers the broader repo surface

Even if `ttsc` becomes a good TS validation path, it does not replace the repo's existing need to format and lint:

- TypeScript and JavaScript
- `json` and `jsonc`
- generated output and workflow-adjacent files touched by CI

So the TS7 promotion does not imply removing Biome.

### The validation lanes are not strictly apples-to-apples

The current comparison is still useful, but the commands do different work.
That means the speedup should be interpreted as “promising enough to keep evaluating,” not as final justification for replacing the root lane.

## API Docs Pipeline Notes

The old TypeDoc/Starlight path has been removed from the active docs runtime. The relevant question now is how confidently the repository can keep the maintained API docs pipeline on the default TS7 compiler line while evaluating faster alternatives around it.

### 1. API Extractor + API Documenter

This is the current pipeline, and it remains the strongest fit for published package contracts.

Pros:

- stable API report model
- good fit for published package contracts
- official markdown generation path
- currently works with the promoted TS7 workspace output in this repository

Tradeoffs:

- still depends on programmatic compiler/tooling compatibility
- still deserves monitoring because it uses its own bundled compiler/tooling stack

### 2. Custom compiler-driven generation

A custom `typescript` or `ts-morph` pipeline is possible, especially if the repo wants tighter control over package docs and generated MDX.

Tradeoffs:

- highest maintenance cost
- most custom code to own
- hardest path to justify while the current API Extractor pipeline is serviceable

### 3. Keep the default compiler flow and treat `ttsc` as additive

This is the current repository posture.

It preserves:

- the existing docs site integration
- the current API Extractor flow
- the checked-in docs generation behavior
- the broader package build/declaration pipeline

If the repo wants `ttsc` to take a larger role, the next step is to decide where it should augment the workflow: contributor fast path, CI comparison lane, or a future default no-emit check.

## Recommendation

Current recommendation:

1. Keep the isolated TS7 validation lane.
2. Keep the default root compiler flow on TS7.
3. Keep Biome as the repo-wide formatter/linter.
4. Use `ttsc-graph` as a supported exploration tool for architecture and runtime-flow questions.
5. Revisit whether `ttsc` should replace or supplement any default checks only after packaging and API-docs tradeoffs are clearer.

In short: the repository is now on TS7 by default, and `ttsc` remains valuable as a fast comparison lane rather than a full replacement for the current root workflow.
